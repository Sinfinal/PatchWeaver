# 赛题背景

云计算基础设施中，内核漏洞（CVE）修复需要兼顾“及时性、稳定性、不中断”。

内核热补丁技术（livepatch）可在不停机条件下修复内核缺陷。比如，社区常用的 `kpatch` 工具集可以完成内核补丁的构建和管理，但其对补丁修改有诸多限制，例如：

- 不能修改初始化函数
- 不能修改静态分配数据
- 不能修改缺少 `fentry` 调用的函数

这导致上游社区的修复补丁并不总能直接转化为可加载的热补丁。

本赛题以“智能体（Agent）”为核心，要求参赛者构建一个面向上游 CVE 修复补丁的自动化系统，完成以下目标：

- 自动获取和解析补丁
- 理解修复意图
- 在保持语义等价的前提下改写补丁，使其满足热补丁机制约束
- 形成可验证的热补丁产物

最终目标是提升漏洞修复闭环的效率与成功率。

# 赛题任务

参赛者需要实现一个内核 CVE 热补丁自动生成智能体，对给定的上游 CVE 修复补丁集合进行自动化处理，利用 `kpatch` 工具链产出可加载的 `livepatch` 内核热补丁，并提供可追溯的过程与验证结果。

任务内容包括：

1. `CVE 查询与补丁获取`
   输入为上游 CVE 编号集合。Agent 需要查询 CVE 数据源，定位上游修复提交，确定针对目标内核版本的补丁，并获取 `patch` 文件。
2. `自动改写与适配（核心）`
   在不改变修复语义的前提下，对上游 `patch` 做结构化改写，使其满足 `kpatch` 构建约束，并能在目标源码树上成功应用与编译。允许多轮迭代，基于构建错误日志自动驱动下一轮改写，直至成功或达到最大重试次数。
3. `自动构建与热补丁生成`
   自动调用 `kpatch-build` 工具构建内核热补丁模块。对失败原因进行归因分类，至少能区分 `patch apply` 失败、编译失败、`kpatch` 限制等，并驱动下一轮改写。
4. `输出产物与报告`
   输出结构化报告，包括每个 CVE 的尝试次数、最终结果（成功/失败）、构建日志、改写前后的 `patch` 文件、构建产物等。

赛题还要求尽可能保证结果可复现，即同一输入在相同环境下可得到一致结果。

预期技术指标包括：

- `热补丁生成成功率`：在规定的 CVE 修复补丁集合上，成功生成并通过加载验证的比例达到 `60%` 以上。
- `语义一致性`：改写补丁保持与上游修复意图一致。
- `效率指标`：平均每个补丁的尝试轮次不超过 `5` 次。

# 赛题特征

## 目标内核与工具链

- 使用龙蜥操作系统（Anolis OS）的 ANCK 内核为目标内核
- 使用 `kpatch` 工具链构建 Linux `livepatch` 机制支持的热补丁模块

## 测例与数据集设计

- `公开数据集（Challenge）`
  来源于“上游社区针对目标内核版本的 CVE 修复补丁”，用于智能体开发过程中打通流程，覆盖典型难点，包括：
  - 删除静态局部变量导致对象文件的 `section` 发生变化
  - 编译器优化导致函数内联，进而引发 `init` 函数变化
  - 修改数据结构定义导致函数 CRC 校验变化，进而引发 ABI 变化
  - 修改缺少 `fentry` 调用的函数导致无法 `hook`
  - 修改全局数据导致 `unreconcilable difference`
- `最终评测集（Final）`
  来源于“最新上游针对目标内核版本的 CVE 修复补丁”，用于最终排名，避免针对性调参。

## 交付产物

- 百炼平台的应用链接
- 代码仓库
- 完整的文档说明，包括详细设计、部署方法、使用说明等

## 验收方式

- `构建验收`
  修改后的 `patch` 可通过 `kpatch` 工具构建为热补丁模块。
- `运行验收`
  模块可加载和卸载，并通过功能验证和回归测试。
- `结果产出`
  每个补丁都需要提供结构化 `JSON` 报告、日志和生成产物。

## 智能体要求

- 推荐使用百炼平台搭建智能体应用，并使用函数计算 FC 创建 MCP 服务，调用 `kpatch` 工具构建内核热补丁
- 推荐使用 `Qwen` 系列大语言模型

# 评审要点

## 核心效果：热补丁生成成功率（50%）

在最终评测集上，成功生成并通过加载验证的数量和比例，是主排名依据。

## 正确性与安全性（30%）

- 改写保持修复语义
- 不引入回归或风险操作
- 验证步骤可靠

## 工程完整度与可复现性（10%）

- 环境支持一键部署（容器或脚本）
- 输入输出规范
- 可在评测环境稳定运行

## 文档与 Demo 表现（10%）

- `README`
- 架构说明
- 测试说明
- 失败案例分析
- 录屏 Demo 清晰可复现
